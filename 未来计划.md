## CAPCW 核心引擎现状（2026-06-16）

CAPCW（内容寻址预测编码工作空间）是 FE-LLM 第一个有系统实证的核心引擎方向，已从受控合成接到
**真实 controller 活文本主动推理闭环**：内容寻址绑定/induction ✅、接回 controller（工作记忆+
surprise→动作+grounded 生成+主动推理 surprise 平复）✅、多跳推理（关键=潜在 CoT，decode→re-embed+
中间监督）✅(机制)、**多跳 CoT 接回 controller 对内多步推理（链式取回+可溯源 CoT trace）✅ + 活文本多跳闭环
（复合所有格→链式取回→grounded+CoT trace，"A的经理的工位"→答）✅(2026-06-16)**、自我成长（穷则变机制成立
但小 WM 不划算）🟡、小 d 稳定性已稳 ✅。多跳「关键」经 2×2 析因纠偏=**中间监督**（非 decode→re-embed）✅；中间监督**可自举**（单跳 teacher 自生成
中间步、免 GT 中间标签，达 GT 的 97%）✅。唯一硬边界=小 d 容量（按容量纪律暂不增 d）。详见

- **小 d 绑定稳定性诊断（完成，2026-06-15，重要纠偏）**：对照诊断"d=32 高方差"。结论：高方差实为 **net 初始化未固定种子的 bug（已修）**——固定后 base(iters=3) 又稳(std≈0.02-0.04)又高(0.82-0.93)；**更多弛豫步(iters=6)灾难性破坏绑定(崩≈随机)**，iters=3 是甜点；高负载均值随 K 下滑是 d=32 容量受限固有特性。`capcw_binding_stability_eval.*`，构想第 22 节。教训：归因训练不稳定前先排除种子未固定；弛豫步要节制。

- **多跳链式破解（完成，2026-06-15，重要正面）**：连试三版新机制。**方案 A 写回重弛豫/B 逐跳深度**（`capcw_multihop_v2_eval.py`）多跳增益 +0.0015 → FAIL（潜迭代读变体，没触根因）；**方案 C decode→re-embed + 中间监督（潜在思维链）**（`capcw_multihop_cot_eval.py`）H=2 cot 0.471 vs e2e 0.104、增益 **+0.2991 → PASS(机制)**。**结论：多跳链式的关键是显式解码中间符号再据它检索（CoT），非潜空间反复 attention——独立复现"LLM 多跳要 CoT"**。绝对值受 d=32 容量限制。构想第 23 节。

- **多跳 CoT 接回真实 controller（完成，2026-06-16，集成里程碑，提交 e2d1149）**：照搬单跳成功路径（引擎 eval→工作记忆适配器→controller API）把第 23 节机制接回 controller 的**对内多步推理**。新组件 `active_inference/capcw_chain_memory.py::CAPCWChainMemory`（与单跳并列、零回归）：key/value 共享 `sym_emb`（取回 value 可作下一跳 key=链式前提），`decide_chain(start,H)` 首跳 surprise→ASK/ANSWER + decode→re-embed 逐跳取回链尾 + `chain`=可溯源 **CoT trace**；controller `capcw_chain_memory_path` 可选加载 + `chain_working_memory_decision`（默认关、与单跳 `capcw_memory_path` 并存）。集成 eval `capcw_multihop_controller_eval.py`（三臂 cot/baseline 单跳/latent 消融）：ASK/ANSWER balacc **0.889**、多跳(H≥2) cot 链尾取回 **0.912**、cot−baseline 任务成功 **+0.392**（链式增量）、**cot−latent(held-out) +0.228** → **PASS**。1 跳 sanity（cot==baseline）通过。教训：判机制必要性看 held-out 泛化（干净 pair 上潜读出 train 不崩但 held-out 输 cot）；新组件并列+默认关=零回归最稳。构想第 24 节、`经验.md`。

- **活文本多跳闭环（完成，2026-06-16，真实系统集成）**：把多跳从显式 API 推进到**活文本自动多步推理**（与单跳 §19→§20 同构）。`MultiHopBindingNLU` 解析复合所有格「X的R1的R2…是多少」(n≥2)→base+rels；`CAPCWChainMemory.decide_path_str` 字符串层 **decode→re-embed** 逐跳取回（cur=base，逐跳 "cur的r" 取回→cur=value，断链→ASK）；接进 `respond()`（默认关、`incontext_chain` 暴露 CoT trace、grounded 回答）。eval `capcw_multihop_dialogue_eval.py`：脚本「记住项目甲的经理是张三/记住张三的工位是B302 → 项目甲的经理的工位是多少 → 答 B302(trace 张三→B302)」；断链(surprise 1.0)→补边→再问→答(surprise 0.121，主动推理平复)；不劫持寒暄。聚合 决策 balacc/链尾取回/劫持 = **1.0/1.0/0.0** → **PASS**。核心：「对内多步推理」在活文本路径成立，**字符串层"解码中间符号→拼下一属性→再检索"是「多跳要 CoT」在活系统最直接落地**。教训：d=32 双跳 ~10% 误检索，单测用确定性种子 + 干净分会话（断链用例第 2 跳全无绑定→必 ASK）。全量 **193 全绿**。构想第 25 节、`经验.md`。

- **纯涌现潜 CoT 2×2 析因（完成，2026-06-16，重要纠偏）**：问"不教中间符号，decode→re-embed 本身能否让链式涌现"。2×2(读出 decode/latent × 监督 中间/仅末端)。多跳(H≥2)：e2e 0.114 / latent_is 0.374 / **emergent(decode+仅末端) 0.132** / cot 0.413 → **emergent≈e2e，纯涌现 FAIL**。主效应：**中间监督 +0.271 ≫ decode→re-embed +0.029**。**纠偏：第 23 节「多跳关键是 decode→re-embed」修正为「关键是中间监督（监督/示范中间步），decode 只是次要增量」**（latent+中间监督 0.374 已近 cot 0.413）。脚本 `capcw_multihop_emergent_cot_eval.py`，构想第 26 节、`经验.md`。

- **自蒸馏中间步（完成，2026-06-16）**：承接"中间监督是主因"，问"现实只有最终答案标签能否自举中间监督"。读出统一 decode→re-embed，唯一变量=中间步来源(无/自生成/GT)，最终跳一律 GT；teacher=单跳取回(只需单跳标签)，自生成中间步=迭代单跳。多跳(H≥2)：final_only 0.576 / **self_distill 0.771** / gt 0.793 → **self 达 GT 97%、比 final +0.195 → PASS：中间监督可自举（免 GT 中间标签）**。且 student≫teacher 迭代(蒸馏放大)；H 越深收益越显。脚本 `capcw_multihop_selfdistill_eval.py`，构想第 27 节、`经验.md`。
