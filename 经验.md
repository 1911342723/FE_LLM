## 终极目标

- FE-LLM 的终极目标不是做一个规则分类器、RAG 外壳或普通 chatbot，而是探索并实现一款基于“能量解释、预测误差、自由能平复、主动推理、可溯源生成、自我成长”的大语言模型。想法文档见FE-LLM道易主动推理方案草案.md，当现实或者有更好的方式时，允许更新该文档，但不允许脱离上述核心概念。
- 当前 `active_inference` v1 只是地基：先把 observation、surprise、policy、expected free energy、trace、memory candidate 等机制做成可验证闭环。任何局部规则、MLP 分类器、teacher 数据清洗，都必须服务于最终的大模型架构，而不是变成新的终点。
- 未来方向要逐步从“外部控制层 + 小模块”走向“模型内部机制”：让模型在生成前先感到 surprise，形成 prediction error，再通过对外检索、对内推理、记忆更新和答案生成来平复自由能。
- 判断一次改动是否正确，不只看当前指标是否上涨，还要看它是否更接近一个可训练、可扩展、可解释、能长期成长的大语言模型。

## 主动推理原型经验

- 不要急着把自由能公式调复杂。先检查 observation 是否能表达 teacher 数据里真正影响 action 的变量。今天 `UPDATE_MEMORY` recall 弱，根因不是 EFE 权重，而是缺少“稳定个人事实/偏好/习惯”的观测信号。
- 中文 prompt 特征抽取要避免单字误触发。“么”“几”作为通用问句触发器会误判“这么叫我”“好几年了”，进而污染 memory、retrieve、clarification。
- 评估报告必须分开看三层：base formula、calibrated formula、final selector。只看最终 action 会掩盖公式是否真的变好。
- teacher 数据是训练材料，不是绝对真理。遇到“你有性别吗？”被标成 `UPDATE_MEMORY` 这类噪声时，不应为了追标签破坏规则边界。
- 可解释公式要先保留变量语义，再做小规模校准。按 action 学习 `risk / ambiguity / action_cost / epistemic_value` 权重，比全局一套权重更符合不同动作的代价结构。
- MLP policy 与 EFE 公式融合时，`classifier_weight` 不是固定经验小数，而是要按 EFE total 的实际尺度和当前数据质量校准。标签清洗前曾需要较大权重；标签清洗后公式更稳，当前默认以 `0.5` 为准。
- `RETRIEVE` 边界不能用“有今天/明天/现在就检索”这种粗规则。应区分 weather、current_time、market_or_news、public_schedule、lookup，并显式排除社交/自我状态问题，例如“你今天开心吗？”。
- 当大量默认转换的 `ANSWER` 样本其实是天气、时间、新闻等外部事实请求时，应先做高置信数据规范化，再训练模型。不要让模型为了匹配错误标签而压低合理的 retrieve 行为。
- `classifier_weight` 会随数据和公式质量变化。清洗标签后，公式本身更稳，默认融合权重应从 `3.0` 降到 `0.5`；否则 MLP 会在 hash fallback 场景下破坏“你好”等基础验收样例。
- 请注意代码要编写中文注释，尤其是涉及公式和权重的部分。这样后续复盘和新成员加入时能更快理解。
- 当你完成终端阶段突破后，推送代码提交

## 多轮闭环经验（实验 B）

- 多轮 surprise 平复的关键不是调公式，而是让 Predictor 形成"可被验证或违背的结构化预期"。上一轮 ask_clarification 即等于预测"下一观测会补充信息"：满足则压低 intent/uncertainty/semantic 误差，违背则把 uncertainty 抬到 0.95。这一条让"对外行动改变环境从而降低未来自由能"第一次有了可测证据（澄清满足后 surprise 平均降幅 >70%，负对照不降）。
- 单轮 teacher 数据训练的 MLP 没有多轮上下文，会把"冲突已被用户纠正"的补充输入仍判成 ask_clarification。解法不是重训，而是给融合层加 `fusion_scale`：多轮证据明确（澄清刚被满足）时置 0，由可解释公式主导。控制层信息多于分类器时，应该削权而不是迁就。
- 记忆读回闭环的最小实现：MemoryManager 启动时读回 jsonl、recall 按"同会话直通 + 跨会话 bigram 重合"召回，注入 belief.assumptions 并让 ActionRealizer 在回答中应用风格偏好。从"只写不读"到"影响下一轮输出"，成长闭环 0→1。
- retrieve 融合权重的根因：teacher 标注高 uncertainty 时 ask 的 EFE 可低至 -0.5+，与 retrieve 差距 ~1.3，统一权重 0.5 永远翻不了盘。按 action 扫描权重（1.2→3.0），2.5 时整体 acc 最高且 precision 仅降 0.011；3.0 开始伤 precision。结论：融合权重要按"公式间分差的实际尺度"定，不是按直觉小数。

## 生成层接入经验（EnergyDecoder × 主动推理）

- 控制层与生成层贯通的关键是共享意图空间：PerceptionEncoder 与 IntentChat 共用同一个 IntentEncoder，信念向量才能直接作为生成目标吸引子。注入时必须做范数重标定——belief 向量经 EMA 平滑后范数偏小，直接混合会让解码器在异常能量尺度下工作。
- 生成质量门控要用能量轨迹而不是只看文本表面：残余能量末值未低于初值即拒绝输出并回退，且被拒绝的轨迹保留在 `rejected_generation` 里，回退本身可溯源。
- 寒暄/身份白名单保留规则快速路径是合理妥协：这类输出属于人格设定而非生成任务，当前字符级小模型对"你好"会生成"没事没事"这类不当回复。其余 answer 一律走能量解码。
- 当前生成层指标（docs/reports/realization_eval.md）：answer 动作 decoder 接入率 100%（11/11），信念注入 11/11，能量整体下降 11/11，单调下降步占比 ~95%，hybrid 选字与纯 logit argmax 的决策分歧率 ~14%。
- 纯 argmin distance 选字在当前训练强度下会退化成循环字（"安。安。安"）：每步贪心最小化能量丢掉语言性。正确做法是草案第 5 节的复合打分 hybrid 模式：score = logP(w) − α·候选内归一化残余能量，α=1.0 时输出连贯且有真实决策分歧。结论：能量信号参与决策的方式是"复合"而不是"替代"。
- 扩大训练（8k→48k 含 LCCC 高熵子集，150 epochs，decode_loss 5.46→1.82）后生成多样性显著改善，但嵌入空间整体变化：①依赖该空间的 policy MLP 必须重训（hidden 64→128、epochs 80 才恢复到 bal_acc 0.929）；②能量量纲变化使相对降幅从 ~35% 缩到 ~0.5%，门控阈值要用相对比较而非绝对降幅。
- 策略 MLP 与编码器是耦合部署的：MLP 在 IntentEncoder 嵌入上训练，hash 回退向量与其分布完全错配，必须按 `encoder_kind` 禁用融合（否则"你好"会被误判成追问）。任何重训编码器的改动，都要连带检查 policy 重训 + 场景验收。

## 翻译泛化实验（opus-100 zh→en）——重要阴性结果

- 用同一套 IntentLM（意图弛豫 + 能量递减解码）在 5 万对 opus-100 短句上训 zh→en，验证"意图空间是否承载跨语言语义"。结论：**当前形态不具备可用的翻译泛化能力**（200 条未见句 mean word-F1 ≈ 0.07，输出集中在少数高频句）。
- 训练范式 bug（已修复，对主架构同样适用）：`IntentLM.forward` 让解码器条件于 z_target=encoder(response)，推理时却只有 z_pred=encoder(prompt)；当 InfoNCE 不收敛（L_int≈ln(batch) 即随机水平）时，解码器面对分布外意图直接塌缩成固定句。修复 = 解码器直接条件于 z_pred（训练/推理同分布 + decode 梯度直训编码器）。修复后从"100% 单一固定句"改善为"疑问句对疑问句、格式模仿、多样化高频句"，但仍远不及格。
- 架构层面的真正教训：**单一 128 维全局意图向量是表达瓶颈**。它足以承载"行动选择层面的粗语义"（policy/对话已验证有效），但不足以独自承载"全句重建所需的细粒度语义"——翻译需要 token 级对齐信息。下一步如果攻翻译/问答，意图必须从单向量升级为序列化/分层结构（道易草案第 6 节"象的层级结构"对应于此）。
- InfoNCE 对齐 zh/en 意图在 8M 字符级模型上基本学不动（L_int 全程 ~5.0-5.5）。跨语言语义对齐需要更大容量或预训练初始化，不要指望小模型从零学出来。

## 槽位化意图判定实验（M2）——第二个阴性结果与转线决策

- 把意图从单 128 维向量升级为 `global+8槽+salience`（意图序列化草案 M1/M2），同数据同规模同预算重测翻译：word-F1 0.0735→0.0832（仅 1.13x），判 FAIL。两次阴性互补锁定结论：**瓶颈不在意图表达结构，而在小模型从零训练的容量/初始化**，按草案预案转预训练底座路线。
- "辅助损失收敛"不等于"能力出现"：槽正交、覆盖损失、salience 全部按设计收敛，输出多样性 67→89、char-F1 0.583→0.642 也确实小幅改善——机制被使用了，但装不下足够信息。判卷只认主指标（word-F1），辅助指标只用来定位机制是否生效。
- 复合能量的分量会竞争"解释生成进度"的角色：加入覆盖能量后，全局能量下降率从 100% 跌到 7.5%，覆盖能量 100% 下降——生成动力学整体转移到新通路。凡是依赖"能量下降"做门控/溯源的代码，换能量函数后必须重新校准用哪个分量（与"量纲变化要用相对比较"是同一类教训）。
- **不能在训练中断点提前判卷**：ep24/80 中断点 checkpoint 自由生成完全坍缩（200 句仅 2 种输出），跑满 80 epochs 后恢复至 89 种。LR 高位（cosine 调度走到 30%）阶段教师强制准确率正常（58%）但自由生成坍缩，是阶段现象不是架构缺陷。同预算对照是判定实验的铁律。
- 槽位结构本身保留：global_intent/intent_slots/slot_salience 作为意图接口的设计（含分槽 surprise、缺槽追问的对接面）在预训练底座路线中复用，被否决的只是"从零训练承载它的 8.9M 小模型"。

## 预训练底座路线 N1（2026-06-13）

- N1 必须先做"冻结底座 + 外挂机制层"的最小可测闭环，不能一开始就下载大模型堆训练脚本。先把 `IntentAdapter`、`EnergyHead`、hybrid score 做成独立可单测接口，才能保证后续接任何底座时机制变量不漂。
- 底座只提供 hidden states 与 token logP；FE-LLM 的贡献只能来自结构化意图、显式残余/覆盖能量、主动推理 trace 与对照组差值。报告中不得把底座自带语言/翻译能力计入 FE-LLM 机制收益。
- P1 判定必须至少保留 A/B/C 三组：同底座无机制、同底座加 FE 机制、随机/打乱 intent 负对照。若 B 不显著优于 A/C，即使绝对指标变好也不能主张机制有效。
- 覆盖能量的工程定义先采用 `Σ_k salience_k · min_{j<=i} ||h_j - slot_k||`，它天然具备前缀非增性质，适合作为"哪些槽位已经被说到"的 trace 量。注意它解释的是覆盖进度，不等同于语言流畅性。
- N1 首发底座选 `Qwen/Qwen2.5-0.5B`，不是因为它最强，而是因为它是 ≤1B、Apache 2.0、中英多语种、base/pretraining 口径更干净。`Qwen/Qwen3-0.6B` 先作为增强复测候选，避免 post-training/思考行为污染机制归因。
- Qwen2.5-0.5B 真实小样本运行前要检查两点：① base tokenizer 可能没有 `pad_token`，封装层需用 `eos_token` 兜底；② Hugging Face 首次请求可能多次超时，但缓存/HTTP fallback 后仍可成功，长任务要持续监控，不要误判为训练失败。当前 32 对、1 epoch 链路已跑通，`rank_loss=3.1931`，checkpoint 约 14MB。
- P1 预测+评估烟测的 1 条样本 FAIL 不能当 N1 结论。当前现象是 A 组 word-F1 0.40，B/C 0.25，且 B/C 输出相同，说明 32 对、1 epoch 的 adapter/head 还没有学出可见机制差异；它只证明 A/B/C 预测 JSONL、eval 报告和判定规则已经接通。
- P1 mini-run（256 对、1 epoch）把 `rank_loss` 降到 1.5253，说明正负配对能量排序目标能被优化。但这仍只是训练损失，不能替代生成评估；下一步必须看 B/C 在多样本预测里是否分化，否则要怀疑"训练目标只约束目标 hidden 能量，未直接优化解码期候选选择"。
- P1 当前排序损失阴性结果：`alpha=1.0` 伤生成（B=0.148<C≈0.149<A=0.161），`alpha=0.2` 退回底座（A=B=C=0.161），候选评分改成"生成前缀平均能量"后只微升到 B=0.153。继续训练到 3 epoch 虽使 `rank_loss=0.4252`，但 B=0.142、C=0.156，说明当前目标学到的是"目标 hidden 匹配"，不是"解码期选对 token"。下一版训练目标必须加入候选级 ranking：同一前缀下 gold next token 的能量低于 top-k 负候选。
- 候选级 ranking loss 是有效方向但不充分：128 对 ex1 让 B=0.208（明显高于 A=0.161）但 C=0.205；128 对 ex4 让 B=0.165、C=0.144（首次 B>A>C）；256 对 ex4 又变成 B=0.177、C=0.187。结论：能量头开始会做"候选 rerank"，但还没有稳定绑定真实 intent。下一步要加 intent 对比：同一 gold/prefix 下，真实 intent 的能量必须低于随机/错配 intent，否则 C 会继续偶然超过 B。
- seed 固定后，intent contrast 的可复现实验结果为 A=0.161、B=0.198、C=0.201。它说明 P1 rerank 能改善底座（B>A），但仍未通过机制归因（B 必须稳定优于 C）。这不是继续扫参能解决的问题：事后能量扣分太弱，下一阶段应做 P1.5——intent-conditioned logits adapter，让真实 intent 在 logits 生成前就进入候选分布。
- P1.5 smoke 首次稳定满足归因方向：128 对、1 epoch、20 条验证，`beta=1.0` 得到 A=0.161/B=0.172/C=0.142，`beta=0.5` 得到 A=0.161/B=0.166/C=0.156。结论：logits adapter 比事后 rerank 更符合"真实 intent 控制生成"的预期，但绝对指标还低，下一步应扩大训练，而不是再回 P1 rerank 扫参。
- P1.5 256 对复测得到 A=0.161/B=0.165/C=0.156，方向仍成立但不如 128 对。不要简单假设扩大样本必然提升；当前必须先做 checkpoint/report 命名管理，保留最佳配置，再跑正式 200 条评估，避免默认 checkpoint 被后续实验覆盖导致结论不可复现。
- P1.5 正式 200 条评估阴性：A=0.182、B=0.158、C=0.175。20 条 smoke 的 B>A/C 没有泛化，说明 logits adapter 仍是浅层候选偏置，不能稳定把真实 intent 注入底座生成过程。P1/P1.5 的共同教训：外挂 rerank/bias 能影响输出，但不够深；下一阶段应按路线草案进入 P2，用 LoRA 或层内 intent-conditioned injection，让 intent 进入 decoder 内部状态更新。
- P2 smoke（IntentResidualAdapter）得到 A=0.161、B=0.172、C=0.172。它说明 residual adapter 能改善底座输出，但 B/C 不分离，当前学到的更像通用 adapter，不是真实 intent 控制。下一步必须强化归因：C 组应支持 batch 内错配 intent，或增加无 intent 的 D 组，避免把通用 adapter 收益误记为 FE 机制收益。
- P2 mismatch control 复测得到 A=0.161、B=0.172、C=0.178。错配 intent 仍不低于真实 intent，说明当前 residual adapter 即使进入 hidden state，也还没把 structured intent 用起来。下一步若继续 P2，训练目标必须显式压低 wrong intent 下的 gold logit，或把 residual 注入更早/多层 hidden；否则会继续学成通用修正器。
- P2 `intent_contrast_weight=2.0` 的分解结果很关键：random control 下 A=0.161/B=0.178/C=0.161，说明模型可以压随机 intent；但 mismatch control 下 A=0.161/B=0.178/C=0.178，说明它压不住相近错配 intent。下一步负例必须从随机噪声升级为 batch 内 hard negative，主判定也应使用 mismatch C 组。
- P2 hard negative 后仍为 A=0.161/B=0.167/C=0.167。结论：只在最后一步 hidden 上做 residual adapter 仍然太浅，训练会学成通用修正或弱扰动，无法稳定使用真实 intent。继续扫权重/负例的收益很低；若继续 P2，需要把 residual 注入多步/多层 hidden，或进入更接近 LoRA 的层内适配。
- P2b 多步训练（candidate_steps=4）仍为 A=0.161/B=0.167/C=0.167，且 disagreement 仅 0.3%。这说明问题不是监督步数太少，而是 shallow 外挂 residual 对底座决策影响太弱/太末端。下一阶段不要继续在最后 hidden 上扫参；要么真正 hook/LoRA 到 decoder 层内，要么回到 `IntentAdapter` 训练目标，检查 structured intent 是否本身表达不足。
- IntentAdapter 诊断给出根因证据：50 条验证样本 `mean_offdiag_cosine=0.8562`，`max_offdiag=0.9715`，slot salience entropy=2.0794≈ln(8)。这说明全局 intent 彼此高度相似，槽位显著性几乎完全均匀；生成端 B/C 不分离不是单纯注入太浅，structured intent 本身也没有形成足够可用的区分。下一步优先修 IntentAdapter：源句区分损失、hard negative contrast、slot entropy/稀疏约束。
- IntentAdapter 分离训练初试（512 对、1 epoch、zh/en contrast + salience entropy + slot diversity）让 salience entropy 从 2.0794 降到 1.4995，说明槽位显著性约束有效；但 mean offdiag cosine 仅从 0.8562 降到 0.8376，global intent 仍高度相似。下一步要强化全局区分，单纯 zh/en 对齐不够。
- 加入 source spread loss 后，512 对、3 epoch 让 mean offdiag cosine 从 0.8562 降到 0.4688，slot entropy 从 2.0794 降到 0.041。global intent 终于明显分离，但 salience 可能过尖；下一步应先用该 checkpoint 回到 P2b 复测 B/C，再决定是否调低熵权重。
- 用 spread IntentAdapter 回到 P2 mismatch control 后，A=0.161、B=0.172、C=0.161，B 再次高于 A/C。这个结果确认根因判断正确：之前 B/C 不分离的重要原因是 intent 表达塌缩，不只是生成注入太浅。下一步应在 spread intent 基础上扩大评估到 200 条，并适度调 salience 熵，避免槽位过尖。
- spread IntentAdapter + P2 residual 的 200 条正式评估为 A=0.1816、B=0.1834、C=0.1823，B-C 只有 0.0011，且 disagreement 只有 ~0.5%。这说明 intent 表达修好后确实有一点方向，但最后一步 residual 对生成决策影响仍太弱，正式集上不能主张有效。下一步要提高生成端影响强度：多层 hook/LoRA，或训练中直接优化更长前缀生成。
- P2c 的第一步不是直接写复杂 LoRA，而是先实现通用 `IntentLayerHook`：按模块路径 hook decoder layer output，把 `IntentResidualAdapter` 的 delta 注入层内 hidden。该工具已用假模型验证 hook 生效且退出后恢复。下一步接真实 Qwen layer smoke。
- P2c layer0 结果：20 条 smoke A=0.161/B=0.213/C=0.177，是目前最强局部归因；但 200 条正式 A=0.1816/B=0.1806/C=0.1780，B 仅高于 C 且略低于 A。结论：层内 hook 方向有效，但 layer0 可能伤语言能力或泛化不足；下一步应做 layer scan（早/中/晚层），不要只凭 layer0 判定 P2c。
- P2c layer scan：layer12 smoke A=0.161/B=0.188/C=0.152，归因更稳且不如 layer0 那么激进；layer23 A=0.161/B=0.167/C=0.175，失败。下一步正式评估 layer12，优先中层注入。
- P2c layer12 200 条正式评估为 A=0.1816、B=0.1772、C=0.1788，smoke 优势未泛化。至此，P1/P1.5/P2/P2c 的最小外挂/单层 hook 路线都未正式通过 N1。下一步若继续底座路线，必须训练真正层内 LoRA/多层注入，而不是推理时外挂 hook。

## V2-M1 判定一（2026-06-13）

- 同预算对照（v1 单向量 IntentLM vs v2 分层 HierarchicalIntentLM，dialogues 3000，同损失同超参）：v1 decode_loss 1.70 / v2 1.88，比值 1.106，**判定 FAIL——分层 PC 反而抬高 next-token decode_loss，且差距随训练增大（12ep 1.037 → 40ep 1.106）**。
- 诚实解读：弛豫把 z_global 压成"去噪的 gist"，牺牲逐字生成所需细节。这是连贯的权衡——分层用 decode_loss 代价换更抽象的意图。
- 因此分层值不值，取决于判定二/三：z_global 是否成了更干净的意图表示（动作分类更准）、分层 surprise 能否定位语义 vs 词形异常。若二/三也无收益，按设计第 8 节停损，回炉 HPC 形式（而不是硬堆层）。
- 工程旁注：发现并绕过一个预存 bug——`tokenizer.build_tokenizer` 的 `from .corpus import all_chars` 在包重构后指向不存在的 `models.corpus`（corpus 实际在 `data/`）；m1 脚本改用按对话字符直建 CharTokenizer。
- 报告：`docs/reports/v2_m1_decode_compare.{json,md}`。
- V2-M1 判定二/三（同一 v2 小预算训练后探针）：
  - 判定二a 意图分工 **FAIL**：z_global action balanced acc 0.182 vs z_local 0.189，两者都≈随机(0.2)。分层没让 z_global 成为更干净的意图表示（有 transfer/欠训练 caveat：编码器在 dialogues 训，探针在 policy_teacher，跨域）。
  - 判定二b 局部分工 平凡通过：z_local token acc 0.70 vs z_global 0.009——构造必然（单向量无法预测逐位置 token），不构成有力证据。
  - 判定三 surprise 定位 噪声级：正常 25.49 / 词序打乱 25.45 / 语义拼接 25.47，差异 ~0.1%，顶层自由能几乎不随扰动变化，无真正定位能力。
  - 综合判定：**当前 2 层 PC 形式不值 decode_loss 代价**（判定一伤生成 + 判定二a无更优意图 + 判定三无定位）。按 v2 设计第 8 节停损：不硬堆层，回炉 HPC 形式或换思路。
  - 报告：`docs/reports/v2_m1_judgments.{json,md}`。
- V2-M1 重定向（动作选择标尺，最干净单变量消融：同一编码器 relax_steps=0 vs 5）：
  - 对照 relax0 best balanced acc 0.9002 vs 实验 relax5 0.9034，**delta +0.003（噪声内）→ FAIL**。policy_teacher 11182 样本。报告 `docs/reports/v2_m1_action_ablation.{json,md}`。
  - 至此分层在三个标尺（生成 / 可解释性 / 动作选择）都未显价值。**根因（重要教训）：当前小标签任务已被单向量表示吃满（动作选择单向量就 0.90，无 headroom），与翻译被底座"填满分数空间"同理。机制（分层）只在单向量装不下的组合性/层级性任务上才会显价值。**
  - 结论：现阶段不强加分层。`HierarchicalPredictiveEncoder` 作为已验证可训练、可弛豫的引擎保留（含完整测试），待出现真正需要层级的任务再启用。当前主线回到已验证有效的 `active_inference`（动作选择/多轮 surprise/成长），把它做深。

## 机制价值首证：belief 追踪有 headroom（2026-06-13）

- 造了一个多轮上下文动作选择任务（同句不同 belief→不同动作：「天气怎么样」已知城市 ANSWER / 未知 ASK_CLARIFICATION；裸城市值「北京」有 pending 则 ANSWER 否则 ASK_CLARIFICATION）。歧义轮占比 0.56。

## 修复 2 个长期 EnergyDecoder 失败 → 全量测试首次全绿（2026-06-13）

## 预训练底座 N2 step1：底座句向量做 policy 特征（2026-06-15，持平偏负）

- 唯一变量对照（句向量来源：自建 8.9M IntentEncoder vs 冻结 Qwen2.5-0.5B），同 split(seed=42)/同 18 标量(teacher surprise_components + 规则特征，与编码器无关)/同 TinyPolicyNet 超参。结果：intent bal_acc 0.9566；backbone mean-pool 0.9385、last-token 0.9346（均 PARTIAL/FAIL）。**4 种 backbone 变体全部未跑赢小编码器**，最好（last-token→PCA128）也差 −0.0070。报告 `docs/reports/backbone_policy_probe.{json,md}`。
- 给了底座最强机会仍输：① last-token（因果 LM 末 token attends 整条序列，比 mean-pool 更合理的句表示）；② PCA 维度匹配（896→128 排除「维度更高/参数更多才赢」的混淆——反而 PCA 版更接近 intent，说明 896 全维对 tiny MLP probe 是噪声/过拟合负担，不是优势）。结论稳健：底座通用句向量在 action 分类上无增益。
- 第三次印证同一根因：**action 分类已被小编码器吃满（~0.957，无 headroom），近饱和任务上「更强表示」没有发挥空间**——与「翻译被底座填满分数空间」「V2-M1 单向量吃满小标签任务」完全同构。判定一个表示/机制是否有价值，必须先确认任务有 headroom，否则对照只会持平。
- 工程经验：probe 要把「公平」做到逐元素——本任务每条样本都带 teacher surprise_components，18 个标量与编码器无关，于是从 intent 特征里切出 `[128:]` 给 backbone 臂逐元素复用，保证唯一变量真的只有 base 句向量。last-token 定位用 `argmax(mask * 位置序号)` 对左右 padding 都鲁棒。backbone 向量按 (seed,n) 缓存为 npz，重跑只训练不再重编码。
- 决策：step1 无 PASS → 不进 step2（不接入 PerceptionEncoder）。要检验底座价值得换到有 headroom 的环节（生成质量/开放理解/更难多轮推理），而不是已饱和的 action 分类句向量。

## 预训练底座 N2 step1b：有 headroom 任务上的表示对照（2026-06-15，重要洞察）

- 承接 step1 推论，把同一「只换表示」对照搬到**有 headroom 的** 9 领域任务 NLU（balanced acc ~0.857，flight/train 措辞重叠未饱和）。同分类器/超参/session 切分，只换 utterance 表示：char-BoW 0.8565、自建 IntentEncoder 0.8387、Qwen mean 0.8537、Qwen last 0.8559、Qwen last→PCA128 0.8565。报告 `docs/reports/backbone_taskdomain_probe.{json,md}`。
- **核心洞察（新增一条判定纪律）：不是所有 headroom 都能被「更强表示」吃掉，要先诊断 headroom 的来源。** 本任务的 headroom 来自领域措辞的**固有歧义**（需更多上下文/数据），不是句向量弱——所以所有表示都卡在 ~0.85，backbone 连最朴素的 char-BoW 都没超过。对比 belief headroom（缺的是 belief/槽位这一**信息**，补上就 0.49→1.0）：那是"信息缺失型" headroom，可被对的特征吃掉；本任务是"固有歧义型"，换表示无效。判定一个杠杆（表示/规模/机制）前，先问"这个任务的 gap 到底由什么造成"。
- 旁证：自建 8.9M IntentEncoder 句向量在本任务反而略低于 char-BoW（−0.018）——压缩句向量丢了部分领域判别的表层信息（领域名/关键字的字符分布），char-BoW 直接保留。提示：领域识别这类靠表层关键字的任务，朴素词袋是很强的基线，别默认"学习式句向量一定更好"。
- N2 总结论：step1（饱和）+ step1b（headroom）两面夹证，**冻结底座句向量在 FE-LLM 控制/理解任务上都没有杠杆**；底座的强项（知识/流畅度）是按容量纪律主动不竞争的方向。底座线再次收敛/封存，主线回自建 v2 控制闭环。教训沉淀：用底座前先确认目标环节的瓶颈确由"知识/流畅度容量"造成，否则只会持平。

## B2 真实任务数据(CrossWOZ)验证 belief headroom（2026-06-15，重要 thesis 修正）

- 在真实人标任务对话 CrossWOZ（清华 thu-coai，5012 训练对话/36910 usr→sys 实质轮）上，用与 `teacher_corpus_eval` 完全一致的口径（同 MLP/split/train，唯一变量=是否加 belief）测「上下文感知 vs 盲」的系统动作预测。脚本 `crosswoz_headroom_eval.py`，报告 `docs/reports/crosswoz_headroom_eval.{json,md}`。
- 适配真实数据的关键发现：**CrossWOZ 系统从不 Request**（用户向系统提问、系统只 Inform/Recommend/NoOffer），所以教师语料的 ask/answer 轴在真实数据上退化。改测两种真有变化的系统决策：offer/nooffer（binary）与 inform/recommend/nooffer（triple）。
- 结果（两方案一致）：belief **不带来 headroom，反而略负**——binary 盲 0.8211→感知 0.8011（−0.0201）、triple 0.7763→0.7654（−0.0110）；且 ambiguous_frac≈0（真实数据几乎没有「同句多动作」）。
- **核心 thesis 修正**：教师合成语料的强 belief headroom（歧义子集 +0.51）**很大程度是构造特性**——它刻意造了大量短的、只有 belief 能消歧的歧义 utterance（占 0.75）。真实任务对话里 utterance 具体、信息量足，系统动作几乎完全由当前 utterance 决定，「belief 给歧义动作消歧」这种 headroom 基本不存在。与 LCCC（开放闲聊 belief 弱 0.655）一脉相承：合成 > 真实任务 > 开放闲聊，越真实 belief-动作 headroom 越弱。
- 诚实边界（不要过度解读为否定 belief）：本实验否定的是「belief 对**系统动作类型**预测的 headroom」。belief/状态追踪对**任务本身**（槽位填充、DB 查询、生成正确回复）仍是刚需——CrossWOZ 全程靠 belief state 运转。弱的是「同句不同动作由 belief 决定」这一特定 headroom，它在真实数据里稀缺。belief 表示也偏弱（只记哪些槽位填了、不记具体值；nooffer 还依赖 DB 实体），但即便如此，结论在 binary/triple 两方案上一致、稳健。
- 方法论教训：把合成 benchmark 的强结论搬到真实数据前，先核对「该数据是否真的具备产生该结论的结构」（这里=同句多动作的歧义轮）。CrossWOZ 的 ambiguous_frac≈0 直接说明真实数据不具备教师语料赖以出强结论的那个结构。

## B2b 真实数据上 belief 的真实 headroom：领域/状态追踪（2026-06-15，正面坐实）

- 承接 B2，给 belief 找它在真实数据上**真正有 headroom 的环节**。任务=预测用户当前轮领域（餐馆/酒店/景点/地铁/出租），blind(utterance 字符袋) vs context(+belief：已出现领域 multi-hot + 上一活跃领域 one-hot)，同 MLP/split/train。脚本 `crosswoz_domain_tracking_eval.py`，报告 `docs/reports/crosswoz_domain_tracking_eval.{json,md}`。
- 结果：总体 blind 0.9248→ctx 0.9776（+0.053）；**领域未明示子集**（utterance 不含目标领域名的跟进句，占 23.5%）blind 0.6638→ctx **0.8552（+0.1914）**。即真实跟进句"营业时间呢/电话多少/怎么去"的领域只能由对话状态决定，belief 在这里把 balanced acc 拉高近 0.2。
- **B2+B2b 合起来精确定位 belief 的真实价值**：belief 在真实数据上对**动作类型**无 headroom（B2 −0.02），对**状态/领域追踪**有强 headroom（B2b +0.19）。结论不是"belief 在真实数据上没用"，而是"belief 的用武之地是状态追踪/指代消解/领域跟踪，不是动作类型选择"。这同时修正了论点又证明了机制在对的环节确有价值。
- 工程教训：CrossWOZ 的 General 意图 act 形如 ['General','greet',...]，act[1] 是 greet/thank/bye 而非任务领域；取领域必须先排除 act[0]=='General'，否则寒暄子意图会污染领域标签空间（首跑就踩了这个坑，修复后领域从 8 个噪声类回到 5 个真领域，未明示子集 delta 从 +0.128 升到干净的 +0.191）。

## B2c belief 对回复内容(grounding 生成)的 headroom（2026-06-15，价值地图收口）

- 把 belief 价值延伸到「系统该说什么内容」：预测系统主告知的 (领域·槽位)（如 餐馆·营业时间，33 类标签），blind vs +belief。脚本 `crosswoz_response_content_eval.py`，报告 `docs/reports/crosswoz_response_content_eval.{json,md}`。
- 结果：总体 blind 0.8165→ctx 0.8977（+0.081）；**领域未明示子集**（跟进句，24.5%）blind 0.6707→ctx **0.8250（+0.154）**。跟进句里槽位词常在 utterance（"营业时间呢"）、领域不在，belief 补领域→系统说对内容。
- **B2 系列完整 belief 价值地图（真实数据 CrossWOZ）**：
  - 动作类型选择（B2）：belief **无** headroom（−0.02）——真实数据动作由 utterance 决定；
  - 状态/领域追踪（B2b）：belief **强** headroom（未明示子集 +0.19）；
  - 回复内容 grounding（B2c）：belief **强** headroom（未明示子集 +0.15）。
- 一句话结论（贯穿 B2/B2b/B2c）：**belief 在真实数据上的用武之地是"谈什么"（状态追踪/领域跟踪/指代消解/内容 grounding），不是"怎么回应"（动作类型，已被当前 utterance 决定）。** 这把"belief 决定动作"的合成论点修正为更准的"belief 决定语境/内容"，且在真实数据上正面坐实了机制价值。

## B2d 把"活跃领域追踪"接回真实 controller（2026-06-15，洞察落地）

- 把 B2b/B2c 的真实数据洞察（belief 价值在领域/状态追踪）落进真实 controller：`BeliefState` 加 `active_domain`（跨轮持久）；`observation` 暴露本轮 `domain`；`belief_update` 识别到领域则更新、否则沿用上一活跃领域；`controller._apply_slot_belief` 加 B2d 分支——对"领域未明示的跟进句"用活跃领域的必需槽位跨轮检查完整性。
- 新能力（此前 controller 做不到）：多槽位领域**不必每轮复述领域**即可主动补全。例：「帮我订酒店」→ASK；「北京」(仅给 city，不复述)→controller 凭 active_domain 知道还缺 date→ASK；「明天」(仅给 date)→ANSWER。此前必须每轮重说「帮我订酒店」才会重新检查（见 `test_hotel_multi_slot_requires_all_slots` 旧行为）。
- 安全落地（行为变更但零回归）：窄门控仿 learned-NLU 三重保险——仅当「有活跃领域 + 本轮提供槽位值 + 无 weather/记忆/安全/逻辑冲突/寒暄信号」时才接管，否则不动。新增 2 个回归测试（主动补全 + 寒暄不被劫持）。**全量 121 测试全绿**（119→121，零回退）。高风险改已验证 controller 前走了 `request_approval` 拿到用户明确批准。
- 教训：把离线 eval 的洞察（B2b/B2c 测得 belief 在领域追踪有 headroom）落成在线能力时，最稳的路径是"加法式新字段 + 窄门控接管 + 新增测试锁定 + 全量保绿"，而不是改既有分支的判定逻辑。门控信号直接复用感知层已有的 weather/记忆/安全/寒暄标志，与既有 learned-NLU 门控同源，保证不劫持既有轮。

## B2e 多域切换 + 修复 B2d 被学习式 NLU 架空的隐藏 bug（2026-06-15，验证救场）

- 顺着未来计划做 B2e（活跃领域追踪扩到领域切换）时，先写了多域会话快速验证（订票→给route→订酒店→给city→给date），**幸亏先验证**：发现 B2d 在领域切换下失效——"上海"（订酒店后只给 city）竟回到 active_domain=booking 且 answer（应 hotel + 追问 date）。
- 根因：学习式 SlotIntentNLU 对**裸槽位值**高置信误分类（"北京"/"上海"→booking 0.99、"明天"/"后天"/"8点"→reminder 1.0），于是裸值轮被设了错误 domain + required_slots。后果有二：① 它覆盖了 active_domain；② B2d 分支只在 required 为空时触发，而 NLU 总把 required 填上，导致 **B2d 几乎从不真正生效**——之前单域 B2d 测试"对结果错原因"通过（靠 NLU 误判 + 槽位巧合，不是靠 active_domain）。
- 修复（仍加法式）：① `observation` 用 `keyword_domain`（仅关键词检出的可靠领域）作为 active_domain 唯一来源，NLU 对裸值的猜测不再更新活跃领域；② `is_bare_slot_value` 收紧为"给了槽位值且无任何领域关键词(含 reminder)"；③ controller 把 B2d 分支**前置**于 required 分支，对裸值跟进句用活跃领域语境覆盖 NLU 的误判 required。修复后多域切换正确：订票→订酒店→"上海"(挂 hotel,追问 date)→"后天"(hotel 凑齐,answer)。新增 B2e 端到端多域切换测试。**全量 122 测试全绿**。
- 教训（双重）：① **落地洞察后必须用真实多步会话验证，别只靠单元测试**——单测可能"对结果错原因"地通过，掩盖机制没真正生效。② 学习式模块（NLU）对**分布外输入**（裸槽位值不是它训练的请求句）会高置信乱判；把它的输出接入决策时，要让**可靠的规则信号（keyword）在语境类决策上优先**，把学习式限定在它擅长的范围（请求句意图），否则它会越权污染状态。

## 分层预测编码世界模型 · 组合泛化裁决 → 正式封存（2026-06-15，蓝图核心定论）

- 起因：用户尖锐质疑"分层和有没有 headroom 关系大吗？为什么要硬找一个能出来的任务？小任务做不到岂不是没用？"。这个质疑是对的——硬找 headroom 任务证明分层 = motivated reasoning（先有结论再凑任务，与 B2 构造特性同坑）。
- 校正后的**原则性裁决**：分层/结构在理论上唯一被认为不可替代的好处是**组合泛化**（见过 A×X、B×Y，泛化到未见的 A×Y）。所以不凑 headroom，而是直接测这个好处。脚本 `world_model/compositional_generalization_eval.py`：任务 compare（A<B/==/>，3 类，可正常训练学会、必须同时知道 A 和 B、标签跨组合复用），训练只见部分 (A,B) 组合、测未见组合；唯一变量=分层机制 flat(relax0) vs hierarchical(relaxK)，同架构同预算同特征维。
- 结果（3 seed，未见组合 accuracy，随机基线 0.333）：flat **0.652±0.083**、hierarchical 0.637±0.123、hier_full(concat) 0.600±0.082；best_hier − flat = **−0.014 → FAIL**。逐 seed 互有胜负、噪声不小，但均值上**分层 ≤ 扁平、无一致优势**。报告 `docs/reports/compositional_generalization_eval.{json,md}`。
- **定论（回答用户的质疑）**：分层在本规模"没用"不是"还没找到对的任务"——而是**即便在它的理论主场（组合泛化），它也没赢扁平**。结合 V2-M1 判定一（分层还要付 decode_loss 代价 + 判定二/三无收益），分层世界模型在 FE-LLM 现规模是**过度设计，正式封存**（引擎+测试保留作存档，像底座线一样）。项目价值锁定在已验证的主动推理控制层。
- 方法论教训（重要）：判定一个架构/机制"有没有用"，不要去找"能让它出彩的任务"（motivated reasoning），而要直接测它被理论声称的**那个唯一好处**（这里=组合泛化），并设公平对照（唯一变量+同预算）。赢→真有用；平/输→诚实定论是过度设计。这比"再换个任务试试"高效且不自欺。
- 诚实边界：结论是"无可证明的优势"（足以判过度设计——分层必须证明它值 decode_loss 代价），噪声下不强主张"分层一定更差"；也不排除"更大规模/更长训练(grokking)/更深层数"下分层另有故事，但那超出 FE-LLM 容量纪律，不在本项目 scope。

## 核心引擎方案一 CAPCW · 绑定任务验证 → 第一个正面核心引擎结果（2026-06-15）

- 分层世界模型封存后，按"借鉴 Transformer 真正力量=内容寻址路由"提出方案一 CAPCW（内容寻址预测编码工作空间：世界状态=一组可寻址 slot，误差按内容路由到 slot，弛豫，穷则变生长）。先定任务+判据再写引擎（不重蹈"先建引擎再找任务"）。
- 任务=in-context 键值绑定（每样本随机 K 对 key→value，问某 key 的 value，单向量可证不足）；唯一变量=世界状态结构。脚本 `world_model/capcw_binding_eval.py`，报告 `docs/reports/capcw_binding_eval.{json,md}`。
- **容量依赖是关键发现**：① d=64（容量充裕，烟测）单向量就能装下 K≤5 绑定（flat≈0.99），CAPCW 反而略逊（slot-attention 难训）——差点据此误判 FAIL；② d=32（容量受限=FE-LLM 实际处境，2 seed）单向量崩（flat 0.43~0.76），CAPCW 决定性胜（0.84~0.99），高 K 平均 +0.29 → **PASS**。已封存分层近随机（0.15~0.33）。
- **教训（重要）**：判定"内容寻址 vs 单向量"必须在**容量成为瓶颈**的区间——容量充裕时单向量够用、看不出差别（又是 headroom/测量问题，但这次在容量轴上）。先用一次 smoke 误判 FAIL，换到容量受限区间才看出 CAPCW 的真价值。这不是"凑 regime"：容量受限正是 FE-LLM 容量纪律下的真实处境，是代表性而非 cherry-pick；且同时报告了 d=64 的反面，诚实。
- 与分层的对比意味深长：分层连理论主场（组合泛化）都赢不了扁平 → 封存；CAPCW 在它的理论主场（容量受限的绑定）决定性赢 → 第一个正面核心引擎方向。区别在于 CAPCW 用的是 Transformer 已验证有效的**内容寻址横向结构**，分层用的是**固定纵向结构**。
- 诚实边界：本轮只验证了"内容寻址工作空间 > 单向量"这一核心机制（slot-attention 形态）；CAPCW 的"自由能解释/可溯源/穷则变生长"完整形态尚待逐步落地 + 各自判定，不可一次性宣称全部成立。
- **阶段二a（2026-06-15）把 slot-attention 升级为显式自由能形态（PCWorkspace）**：slot=解释输入的混合成分、生成 g 预测输入、路由由重建误差导出（attention 即推理）、弛豫沿 -dF/ds 下降、可溯源（responsibilities/error/F-trace）+ 穷则变生长钩子。单测锁定（自由能下降/责任归一/更多 slot F 更低/生长钩子）。判据=PC 形态不丢绑定胜势：d=32 高 K 平均 CAPCW_PC−flat≈+0.27，与裸 slot-attention 持平、远超单向量 → PASS。即把黑盒 attention 升级为"显式预测编码/自由能/可溯源"**不损失内容寻址能力**。全量 130 测试全绿。教训：给已验证的"内容寻址"机制补上 FE-LLM 身份（自由能/可溯源/生长）时，用"判据=不丢已验证能力 + 新性质单测（F 下降等）"双保险，确保 elaborate 没把核心能力做没。

## CAPCW 阶段二b：穷则变（结构成长）判定（2026-06-15）

- 穷则变「自由能高→加 slot」要成立，前提是 slot 数=绑定容量。脚本 `world_model/capcw_growth_eval.py`：固定 K=5、d=32 扫 slot 数 M（CAPCW_PC accuracy）：M=2 0.397 / 3 0.649 / 4 0.754 / 5 0.885 / 7 0.893。M<K 平均 0.60 → M≥K 平均 0.89，转折 +0.29 → PASS：**slot 数确实是绑定容量**，长到 M≥K 才解。grow 钩子能检测欠容量。
- 诚实 caveat：本实验固定 M 训练扫出"容量 vs slot 数"曲线（验证穷则变前提）；grow 钩子停止阈值需校准（本次生长到上限 8 而非恰好 K=5）；训练期动态生长完整闭环是后续。
- 阶段二小结：CAPCW 三步落地——内容寻址>单向量（一）、显式自由能/可溯源不丢能力（二a）、slot 数=绑定容量→穷则变前提（二b）。

## CAPCW 阶段三：slot 工作空间接控制层，价值在内容取回（2026-06-15，与 B2 互证）

- 把 slot 工作空间接到动作决策。脚本 `world_model/capcw_action_eval.py`：绑定+双输出（动作类型 ASK/ANSWER/REFUSE 粗 + 回复内容=精确 value 细），d=32。结果：动作类型 flat 0.556/capcw 0.911、**回复内容·精确 value flat 0.443/capcw 0.829（+0.386）** → PASS。query→slot 路由分离 bound/unbound≈0.48（surprise 信号）。
- **深意（两条独立证据互证）**：CAPCW 在控制层的价值落在**内容/状态取回**（精确 value，单向量做不到），不在粗动作判断——这在合成引擎层独立复现了 B2 真实数据的结论"belief/状态价值在内容/语境、不在动作类型"。真实数据(B2) + 合成引擎(CAPCW Phase3) 指向同一结论。
- 实验设计教训（再次）：第一版纯「成员判断」动作（query 在不在场→答/问）单向量也能 ~1.0、无 headroom（连试三个太易任务才悟到）；把动作改成 value 依赖 + 与 value 头联合训练，才在"内容"维拉开。判别机制价值，关键永远是"让任务的瓶颈正好落在被测机制上"。
- CAPCW 四判定全正面：内容寻址>单向量(一)/显式自由能·可溯源不丢能力(二a)/slot 数=绑定容量→穷则变前提(二b)/接控制层价值在内容取回(三)。下一步：推向更贴近语言的真实任务 + 接回真实 controller（grow 校准 + surprise 门控 + 条件生成）。

## CAPCW 三部（推向真实 + 控制闭环，2026-06-15）

- **Part 1 真实语言(CrossWOZ)绑定——价值边界精确化**：用真实 CrossWOZ (领域·槽位→值) inform 绑定检索，flat vs CAPCW。real（相关/可记忆）flat 0.981 ≫ capcw 0.668；同一份槽/值 shuffle 成真 in-context（值不可由键预测）→ flat 0.297 ≪ capcw 0.883（+0.586）。**结论：CAPCW 的内容寻址优势专属真 in-context 绑定；真实 CrossWOZ 槽值可记忆→单向量记先验就赢、不需要它。定位=工作记忆/in-context 绑定引擎（指代/induction/新关联），非记忆型世界知识**（恰是 Transformer attention vs MLP 分工）。教训：把机制推向"真实数据"前先确认真实数据是否真有该机制所需的结构（这里=in-context 不可记忆性）；否则真实数据反而因可记忆而让简单方法赢。
- **Part 2 穷则变自校准 + 按需分配**：相对边际增益准则（加 slot 不能再降自由能 ≥min_rel_gain 就停）；训练 max_slots、推理按 K 选 grow_m：K=2→3.5 / 4→6.5 / 6→7.0（随 K 增），acc@grow≈acc@max（损失≤0.04）→ PASS。比阶段二b"长到上限"前进。脚本 `capcw_grow_dynamic_eval.py`。
- **Part 3 surprise→动作闭环（无动作监督）**：只学绑定取值、从未见 ASK/ANSWER，仅用 query 路由 surprise 阈值判动作：seed43 绑定 0.90→闭环 balacc 0.881；seed42 绑定 0.49→0.571；均值 0.726(PARTIAL)。**机制成立（绑定良好时 0.88 无监督分开"该答/该问"），限制=d=32 绑定训练高方差**。脚本 `capcw_surprise_action_eval.py`。教训：surprise→动作 是涌现的（不需单独动作头），但质量受底层绑定学习质量主导。
- **CAPCW 总评**：从受控合成（四判定全 PASS）走向真实/控制闭环——方向正确、边界清晰（in-context 绑定引擎）、限制诚实（绑定训练方差/真实数据可记忆性）。是 FE-LLM 第一个有系统实证的核心引擎候选。

## CAPCW induction 负结果：内容绑定 ≠ 序列相邻（2026-06-15，能力边界）

- induction（...A B ... A→B，in-context learning 基石）：flat 0.118 / CAPCW 0.104 **双双接近随机**（基线 0.05）→ FAIL。脚本 `capcw_induction_eval.py`。
- 诊断：induction 需"相邻关系"（cue 后紧跟的 token）；当前 CAPCW/flat 都把 token 独立嵌入再池化，bigram a→b 相邻信息聚合时丢失。**集合式 slot 工作空间擅长内容绑定（pair 整体喂入），不自带序列相邻算子**（Transformer 靠 attention+位置实现 induction）。
- 结论/教训：CAPCW 已验证主场=内容绑定/工作记忆（现场寻址检索已配好的关联）；要做 induction 这类"序列内现场学映射"，需在 token→slot 写入前补相邻/序列编码（token 与前驱拼接 / PERBlock 先成 bigram）。这是 CAPCW 走向语言引擎的下一块，应独立判定、不硬凑。诚实标定能力边界比强行让它 PASS 更有价值。

## CAPCW 序列相邻算子：2×2 交互救活 induction（2026-06-15，能力边界突破）

- 承接上面的 induction 负结果（诊断=缺"序列相邻算子"）。给 CAPCW 在 token→工作空间写入**前**补一个最小相邻算子=induction head 的 **previous-token channel**（位置 t 表示 = `proj([emb(前驱); emb(当前)])`，把独立 token 流变成 (prev→cur) bigram 流），重测 induction。脚本 `world_model/capcw_induction_seq_eval.py`，报告 `capcw_induction_seq_eval.{json,md}`。
- **2×2 单变量析因**（相邻算子 on/off × 世界状态 flat/CAPCW，同 d/同 slot 预算；扫负载 n_pairs=2/4/6、2 seed、随机基线 0.05；模型初始化已固定种子→可复现）：只有 capcw_adj（两者兼备）起作用——n_pairs=2/4/6 = **0.642/0.643/0.643**；其余三格全部≈随机（flat_raw/flat_adj/capcw_raw ≈ 0.12~0.18）。rescue_capcw(capcw_adj−capcw_raw) 均值 **+0.4792**；对照 rescue_flat(相邻算子单独喂 flat) 均值 **+0.005≈0**；H2 内容寻址优势(capcw_adj−flat_adj) ≈ **+0.48~0.52** → **PASS**。
- **核心结论（2×2 交互，比朴素假设更深）**：induction 需要"序列相邻算子"与"内容寻址 slot"**两者同时**具备，缺一格都≈随机——相邻算子单独喂单向量池化救不活（无法联想检索）、slot 没有相邻算子也救不活。这把 CAPCW 从"内容绑定/工作记忆引擎"扩到"**in-context 序列引擎**"（指代/induction 这类序列内现场学映射），且**强化了 CAPCW 的核心主张**：内容寻址不是可选项，是 induction 的必要一味。
- 实验设计教训（再次印证"先确认任务有可学性，再看机制"）：第一版沿用旧 induction 的含噪生成（filler 可能撞 cue 造 spurious bigram、bigram 起点重叠互相覆盖），capcw_adj 只到 0.34、被标签噪声压着；清理成与 capcw_binding 同口径的无歧义版（bigram 起点偶数不重叠 + filler 不含任何 a）后 capcw_adj 升到 0.62。清理**对四臂公平、no-adj 臂仍≈随机**，故是去噪不是偏袒。诚实边界：本算子是 prev-token 这一最小相邻基元；更长程/多跳的序列依赖尚未测，留作后续。
- 工程：相邻算子下沉为引擎可复用组件 `capcw.SequenceAdjacency`（与 `PCWorkspace` 并列）+ 3 条不变量单测（形状不变 / 位置0用可学 BOS / 前驱敏感）；eval 复用该组件（单一真相源，避免实现漂移）。**全量 133 测试全绿（130→133，零回归）**。顺修一个 Windows 坑：控制台默认 GBK 编码不了 `≫` 等字符会 `UnicodeEncodeError` → 入口 `sys.stdout.reconfigure(encoding="utf-8")` 兜底（报告文件本就 utf-8）。commit `9e31a48`。

## CAPCW 多跳链式推理负结果：迭代读出 ≠ 链式组合（2026-06-15，边界细化）

- 承接序列相邻算子（1 跳 induction PASS），测**多跳**：序列放链 c0→c1→…→cH（+n_distract 个干扰 bigram），查 c0 答 cH，需现场把多个绑定**链式组合**。脚本 `world_model/capcw_multihop_eval.py`，报告 `capcw_multihop_eval.{json,md}`。
- 三臂（唯一变量=读出结构；三臂共用已验证的序列相邻算子+同预算；**capcw_1read/iter 同 seed 初始化→1 跳严格相等作 sanity**）：flat 单向量 / capcw_1read 单次内容寻址读出 / capcw_iter H 次迭代读出（读出值→to_next→下一跳 query）。hop 1/2/3、3 seed、随机基线 0.05。
- 结果：1 跳 flat 0.100 / capcw **0.423（1read==iter，sanity 通过）**；2 跳 0.059/0.131/0.117；3 跳 0.057/0.079/0.133。H1 iter−flat 多跳均值 +0.067、**H2 iter−1read 多跳均值 +0.020≈0** → **FAIL（诚实负结果）**。
- **核心结论（边界）**：CAPCW 主场是**单跳** in-context 绑定（1 跳 0.42≫flat 0.10）；**多跳链式组合检索不成立**。关键证据 H2≈0——**对固定 slot 反复读（迭代读出）并不能链式组合**（单读≈多读，多跳精度无差）；slot 工作空间比单向量略好（多跳 ~0.12 vs ~0.06）但远不解多跳。
- 诊断：迭代读出每跳读回的是**纠缠的整 slot 向量**，`to_next` 在 d=32 下无法把"中间符号 c_i"干净地再注入成"下一跳的键查询"。试过显式 key/value 读出反而更差（1 跳 0.56→0.32）。多跳链式很可能需要的不是"多读"，而是把中间结果**作为新观测重新写入/重弛豫工作空间**（迭代更新 slot 本身，而非只读固定 slot），或真正的多层深度 + 逐跳学习——这与 Transformer 靠多层做多跳同构，是独立的下一关（应单独定任务+判据）。
- 教训：① **单变量对照要做到"仅差被测变量"**——1read/iter 架构相同（n_reads 只改 forward 读出次数、不增参数），必须**同 seed 初始化**才能让 1 跳严格相等、把多跳 delta 从初始化方差里摘干净（诊断版未同 seed 时 1 跳出现 0.31 vs 0.56 的假"差异"，会误导判读）。② d=32 下绑定/读出训练高方差（1 跳跨 seed std 0.18），结论靠多 seed + 相对 delta，不靠单次绝对值。③ **不为追 PASS 反复换读出结构**（kv 试过更差即止），诚实记负结果 + 诊断比强凑有价值（与 induction 先负后补算子同一纪律）。commit `abc2df5`。

## CAPCW 接回真实 controller：引擎 surprise 驱动"知道何时不该答"+内容取回（2026-06-15，集成里程碑）

- 把已验证的 CAPCW（内容寻址 slot 工作空间 + query 路由 surprise，Part3）做成 controller 兼容工作记忆组件 `active_inference/capcw_memory.py::CAPCWWorkingMemory`：`bind(key→value)` 累积 in-context 绑定、`decide(query)`→引擎 surprise 裁决 `ActionType.ANSWER`(bound,低 surprise,带取回 value)/`ASK_CLARIFICATION`(unbound,高 surprise)。
- **定位（不与 belief 槽位字典冗余）**：`known_slots` 是预定义槽位的精确字典；CAPCW WM 是 **in-context 任意键值绑定**的工作记忆，且 ASK/ANSWER 决策**从引擎 surprise 涌现**（无动作监督）——是"机制从引擎涌现"主张在 controller 上的落地。
- 集成 eval（实验C 同口径，`world_model/capcw_controller_integration_eval.py`）：对话=K 个 in-context 绑定 + 查询(bound/unbound 各半)，FE-agent(CAPCW WM) vs baseline(无记忆·永远直答)。结果：绑定训练 0.894；**ASK/ANSWER balanced acc 0.829**(引擎 surprise，无动作监督)；**内容取回 value acc 0.919**；任务成功率 **0.792 vs 基线 0.043**；unbound 胡答率 **0.255 vs 基线 1.0** → **PASS**。报告 `capcw_controller_integration_eval.{json,md}`。
- **接回真实 controller（加法式、默认关、零回归）**：`ActiveInferenceController(capcw_memory_path=...)` 可选加载；`bind_working_memory`/`reset_working_memory`/`working_memory_decision` 接口；默认 None 不启用 → 既有管线零影响。新增 10 测试（适配器 8 含 save/load + controller 钩子 2：默认关 no-op + 加载后端到端 bound→ANSWER/unbound→ASK），**全量 143 全绿**。
- 诚实边界：① 工作空间需在绑定任务上训练才内容寻址（适用受控小词表）；② 活文本自动把"现场关联"抽成 (key,value) 需一层 **in-context 绑定 NLU**（开放词表/容量受限），属下一步，故现为**显式 API 接口**而非 respond() 自动调用（不夸大为通用文本能力）。
- 教训：把引擎接回 controller 时，先确认增量**不与既有机制冗余**（known_slots 字典 vs in-context 任意绑定），用"加法式组件 + 默认关 + 显式 API + 实验C 基线对照 + 测试锁定"落地；并诚实标注活文本接线所缺的那层(binding-NLU)。commit `f13669f`。

## CAPCW 活文本闭环：in-context 绑定 NLU 接进 respond()（2026-06-15，真实系统集成里程碑）

- 承接"接回 controller"，把它从**显式 API** 推进到**活文本自动**：用户自然语言陈述/查询现场关联，controller 经绑定 NLU 把关联喂 CAPCW 工作记忆，**引擎 surprise** 驱动 ASK/ANSWER + 取回 value。
- 绑定 NLU `active_inference/incontext_binding_nlu.py`（高精度规则，无权重）：bind(记住X是Y / X对应Y / X设为Y / X等于Y / X的{密码|工号|…}是Y)、query(X是多少 / 是什么 / 对应什么 / 等于几)、none(其余)。**查询先于绑定**匹配以消解"X的密码是多少"；**裸"X是Y"与寒暄不触发**（避免劫持，仿学习式 NLU 窄触发教训）。
- 工作记忆加 per-session **str↔id 表**（`bind_str`/`decide_str`）：把活文本任意 key/value 串映射到工作空间符号 id（工作空间学的是**符号无关**的内容寻址路由，故任意串分配不同 id 即可绑定/取回）；未见 key 平凡 unbound→ASK。
- 接进 `respond()`（仿 `context_policy` override、默认关零回归）：capcw_memory 加载时解析本轮文本——bind 存入工作记忆、query 由引擎 surprise 覆盖 ASK/ANSWER；`ModelResponse` 加 `incontext_value`。
- **grounded 可溯源生成（闭环收尾）**：bound 查询的回答**扎根于引擎取回的 value**（"会议室是多少"→"会议室是B302"），bind 给 grounded 确认（"好的，已记住会议室是B302"），unbound 走常规追问文本——回答内容可溯源到引擎绑定，而非通用模板。完成"感知→取回→生成→可溯源"活文本闭环（蓝图"可溯源生成"）。
- **主动推理闭环（surprise 平复，蓝图核心）**：`ModelResponse` 加 `incontext_surprise`（=1-query 路由匹配度，可溯源）。实测闭环——问未绑定"门禁卡是多少"→ surprise **1.000** → 追问；用户"记住门禁卡是8821"补绑定 → 再问 → surprise **0.057** → grounded 回答"门禁卡是8821"。即**对外行动(追问)改变环境(补绑定)从而降低未来自由能**，与实验 B 的多轮 surprise 平复在 CAPCW 引擎路径上同构、可判定（`test_active_inference_surprise_drop_after_clarification`）。
- eval（`world_model/capcw_incontext_dialogue_eval.py`）：脚本会话「记住会议室是B302 / 项目代号对应X9 → 答已绑定取回 B302/X9 → 问未绑定门禁卡→追问 → 你好不劫持」全对；聚合 决策 balacc **1.000**、value 取回 **1.000**、寒暄劫持率 **0.000** → **PASS**。报告 `capcw_incontext_dialogue_eval.{json,md}`。
- 新增 15 测试（NLU 10 + 字符串接口 3 + controller 端到端 2），**全量 158 全绿**。
- 诚实边界：① 绑定 NLU 是高精度规则、覆盖标记式表达（非全开放实体抽取）；② 工作记忆当前是 controller 级**单一工作集**（多会话需 `reset_working_memory()` 或 per-session WM，接口已留）；③ value 词表上限由 n_keys/n_vals 约束，超出需 reset/扩容。
- 教训：把引擎接到活文本时，用"**高精度规则 NLU（窄触发不劫持）+ 符号无关工作空间（任意串分配 id）+ override 接 respond（默认关零回归）+ 脚本demo+聚合双判据**"落地；引擎 surprise 让"知道何时不该答"在真实对话里**从机制涌现**，而非手写 if key in dict。
